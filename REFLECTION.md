## 一、Superpowers 技能：哪些发挥了最大作用，哪些"形式大于实质"？

**最大作用的技能**：

`brainstorming` 是出乎意料的高价值技能。它强制在写任何代码之前，把"做什么"拆成选项让我逐条确认。最初我对"Coding Agent Harness"只有模糊概念，是 brainstorming 的追问让我不得不回答"用户是谁、反馈信号从哪来、危险动作是什么"。这些问题的答案后来直接成了 SPEC 的骨架。没有这个过程，我大概率会跳进代码然后反复返工。

`verification-before-completion` 的"冷启动验证"机制是本项目最有价值的工程实践。理论上写完 SPEC + PLAN 就可以开始实现了，但用"零上下文工程师"视角再审查一遍，发现了 7 个缺陷——其中 AgentLoop 未集成 MemoryStore（P4）和 .harness/ 目录初始化缺失（P6）如果不修，实现阶段一定会出问题。这个技能的价值在于：它不信任"写文档的人觉得清楚了"，而是用陌生视角对文档做压力测试。

`test-driven-development` 在 Mock LLM 驱动的测试中发挥了关键作用。写测试 → 确认红 → 写代码 → 确认绿，这个循环在 AI 协作下反而比手写时更严格——因为 AI 生成的代码可能有你没注意到的假设，而测试就是检验那些假设的客观标准。

**"形式大于实质"的**：

`using-git-worktrees` 和 `subagent-driven-development` 在本项目中并未真正发挥作用。原因是项目规模决定了不需要 worktree 隔离——16 个 Task 虽然逻辑上可分，但实际上是一个连续的依赖链（Task 6 依赖 Task 1，Task 11 依赖前面所有），并行 worktree 的意义不大。subagent 驱动开发在真正的多成员协作场景中很重要，但单人在短时间内完成小项目时，subagent 的启动成本（造 prompt、等结果、审代码）反而高于自己写。

这暴露了 Superpowers 的一个假设：它预设的项目规模和协作模式可能比实际作业场景更重。对小团队（尤其是单人），一些技能更像是"流程上的正确"而非"效率上的最优"。

## 二、TDD 强制在 AI 协作下：阻碍还是放大器？

**是放大器**，但前提是"机制必须是代码"这条规则已经落实。

在 RrratCoder 中，所有核心机制都有 Mock LLM 驱动的单元测试。写 Mock 的过程本身就是在检验"这个机制到底是不是代码"——如果某个功能离开真实 LLM 就测不了，那它就只是一句提示词。TDD 把这一判据操作化了：你先写测试，如果测试写不出来，说明机制设计有问题。

最典型的例子是 Guardrail。写 `guardrail.test.ts` 时，我需要构造 `guardrail('shell', {command: 'rm -rf /'})` 并断言它被拒绝。这个测试完全不依赖 LLM，每次运行结果都确定。反过来，如果护栏是"在 system prompt 里写一句'不要执行危险命令'"，就没有任何测试能确定性验证它——因为结果取决于 LLM 的随机性。TDD 在这里不是"先写测试再写代码"的形式问题，而是"先把机制落实为可测试的代码"的实质问题。

但 TDD 也有代价。对于像 CLI 入口、WebUI server 这种"胶水代码"，mock 测试的投入产出比很低。这些模块的价值在于正确地组装已有模块，而非独立逻辑。对这类模块，我选择了"编译通过 + 集成手动验证"而非强制单测覆盖——这其实偏离了严格 TDD，但我认为合理。

## 三、Subagent 驱动工作流：能自主运行多久？

在本项目中，subagent 的单次 task 执行通常能自主运行 2-5 分钟不出错——前提是 PLAN 中的 task 描述足够精确。

最成功的一次是 Task 6（工具分发器 + 4个工具）：PLAN 中包含了每个工具的完整接口定义、边界条件和安全约束，subagent 照着实现即可，不需要任何额外澄清。最失败的一次是反馈闭环的集成——PLAN 中 `classifyErrors` 调用方式写得太模糊，subagent 误解了输入格式（传入 JSON.stringify 而非原始 stdout），导致分类结果全为空。这个问题最终是我自己发现的，而非 subagent 自检出来的。

结论：subagent 的自主性上限 = PLAN 的精确性上限。一旦 PLAN 有歧义，subagent 不会停下来询问——它会"猜一个合理实现"继续往下走，而这个"合理"可能和你的预期完全不同。这也是为什么冷启动验证如此重要——它是在 PLAN 被 subagent "误解"之前，先把歧义找出来。

## 四、什么样的 Task 颗粒度最优？

从本项目的经验来看，最优颗粒度是：**单一文件 + 单一职责 + 独立可测**。

- 太粗（如"实现反馈闭环"）：一个 task 包含 ErrorClassifier + 3 个 Validator + ValidatorChain + RetryState，subagent 的 context 太长，容易在细节上出偏差。
- 太细（如"写 classifyLintOutput 函数"）：subagent 缺少上下文，不知道这个函数在整体架构中的位置，容易写出与邻居模块不兼容的接口。
- 合适（如"实现 ErrorClassifier，输入校验器名称和 stdout，输出 ClassifiedError[]"）：subagent 知道输入输出契约，可以独立写实现和测试，集成时只需验证接口匹配。

## 五、SPEC 质量如何影响实现质量（具体案例）

最典型的案例是 P9 号缺陷——"反馈闭环触发时机"的措辞不一致。

SPEC §3.5 原本写的是"对 Agent 的每一次工具执行结果进行客观判定"，PLAN 的实现却在 AgentLoop 中加了 `if (['write_file', 'shell'].includes(...))` 的条件判断。如果按 SPEC 的字面意思实现，每次 read_file 和 grep 后也会跑完整的校验器链——这会导致严重的性能浪费（每次读文件都要跑 ESLint + tsc + vitest），而且毫无意义。

这个问题的根因是：SPEC 写"每一次"时，我在潜意识里想的是"每次可能改变代码的操作"，但没有把"可能改变代码"写成明确的约束。拿到 PLAN 后，subagent 正确地判断了"读操作不需要校验"，但 PLANNER（写 PLAN 的 AI）并没有回过来修正 SPEC 中的矛盾措辞。

**教训**：SPEC 中的模糊措辞不会在实现阶段被"自动澄清"——它会被 AI 按自己的理解补全，而不同的 AI（或同一个 AI 在不同 task 中）可能做出不同的补全。SPEC 中的每个"每次""所有""必须"都应该附带边界条件的说明。

## 六、最有效的 Prompt / Context 策略

**最有效的策略：在 PLAN 的每个 task 中写完整的输入输出契约**。

具体做法是：每个 task 的"步骤1"先写测试代码，测试代码本身就定义了接口契约（函数签名、参数类型、返回值断言）。subagent 拿到的 context 不是"请实现一个配置加载器"这种开放式任务，而是"请让以下测试通过"，测试本身就是最精确的需求文档。

例如 Task 2（ConfigLoader）的测试中写了 `expect(config.agent.maxRounds).toBe(10)` 这样的断言，subagent 不需要理解"默认值是什么意思"——它只需要让这个断言通过。这种方式把歧义空间压缩到最小。

**最无效的策略：在 SPEC 中写长篇叙事描述**。像"用户可以通过 CLI 输入自然语言任务"这样的叙事虽然帮助我理清了产品思路，但对 subagent 实现代码没有直接帮助。实现者需要的是"这个函数的输入是什么类型、输出是什么类型、边界条件是什么"，而非"在什么场景下谁会用到它"。

## 七、凭据与分发的工程要求，迫使我想清楚了什么？

凭据安全这条要求，是六个维度之外最重要的工程教育。在 SPEC 中画一个"凭据威胁模型"表格，和实际实现一个 AES-256-GCM 加密存储方案，是两个完全不同层次的理解。

具体来说，我被迫想清楚了三个问题：
1. **Key 的生命周期比存储方式更重要**——Key 从录入、到内存使用、到日志输出、到终端回显、到进程继承、到 Docker 镜像层，每个环节都是泄露面。只加密存储文件但不处理日志中的 Key 泄露，等于只锁了前门开着窗户。
2. **Docker 内的凭据存储是一个"边界瓦解"问题**——Windows Credential Manager 在容器内不可用，AES 加密文件需要主密码交互输入，这两个方案之间不是简单的"降级"，而是安全模型的根本变化（从系统级安全变为应用级安全）。
3. **生产环境的凭据管理远比作业复杂**——本项目的 CredentialManager 只有 150 行，但如果要支持多 Key、Key 轮换、审计日志、权限分级，复杂度会指数增长。这次实现只是摸了冰山一角。

分发方面，Docker 看起来"一条命令启动"，但实际要处理的问题远不止写 Dockerfile——Volume 挂载的权限、首次运行的引导交互、镜像大小优化、多架构构建——每个都是独立工程问题。本项目的 Dockerfile 只做到了"能跑"，离"好用"还有距离。

## 八、如果重做，我会改变什么？

1. **先设计 mock 覆盖矩阵，再写代码**。SPEC 中每个机制应该有一行"此机制需要哪些 mock 单测来验证"，而不是写完代码再回头补测试。这样 PLAN 的 task 分解也会更合理——每个 task 的验收标准就是"通过以下 N 个测试"。

2. **减少 task 数量，增加每个 task 的深度**。16 个 task 看起来全面覆盖了六个维度，但每个维度的实现都偏浅——除了反馈闭环有 6 个文件，其他维度的实现通常只有 1-2 个文件。如果重做，我会砍掉 CLI 和 WebUI（它们本质是胶水代码），把精力集中在让反馈闭环处理更复杂的场景（如跨文件的级联错误、增量校验）。

3. **CI 配置应该在 Task 1 就完成**。本项目 CI 是最后才加的，结果 push 上去就挂了一次（EACCES 问题）。如果 CI 从一开始就存在，每个 task push 时都会自动验证，"本地通过但 CI 失败"的问题会早 8 个 commit 被发现。

4. **SPEC 增加一个"术语表"章节**。在写作和实现过程中，"工具""校验器""反馈信号""护栏"这些词在不同 context 下有不同含义，导致 SPEC 内部措辞不一致（如 P9 的"每一次"问题）。术语表会强制我在写 SPEC 时就明确每个词的精确定义。

## 九、对 Superpowers 方法论的批判

Superpowers 的核心假设是：**软件开发的困难在于流程纪律的松懈，而非智力劳动的复杂性**。它认为只要强制执行 brainstorming → plan → TDD → review → finish 这个顺序，智能体就能产出质量可靠的软件。

在本项目中，这个假设**部分成立、部分不成立**。

**成立的部分**：对于"已经知道要做什么"的工程任务（如实现一个已知接口的 ConfigLoader），流程纪律确实是质量的关键保障。TDD 强制你先想清楚接口再写实现，review 强制你审视 AI 生成的代码是否有隐含假设。这些纪律在 AI 协作模式下甚至比人类独立开发时更重要——因为 AI 不会主动质疑你的需求，它会忠实地实现一个"看上去合理"但可能完全错误的东西。

**不成立的部分**：对于"不知道要做什么"的设计任务（如"反馈闭环应该怎么设计"），流程纪律不能替代领域知识。Brainstorming 能帮你理清思路，但它不能替你做出"三层校验器链 vs 单层全量校验器"这种需要领域经验判断的决策。Superpowers 假设需求方（人）清楚自己要什么，只是需要 AI 帮助执行。但在本项目中，我这个"需求方"对 Coding Agent Harness 的理解是在 SPEC 写作过程中逐步加深的——我最初的理解在很多方面是错的或不完整的。

**最大的张力**在于：Superpowers 的时间线假设（先想清楚再动手）与现实中"边做边理解"的认知模式矛盾。写完 SPEC 时我以为自己理解了反馈闭环，写完实现时发现理解更深了一层，写完测试时又加深了一层。如果严格遵循"SPEC 封板后才能写代码"，那后两层理解就永远无法进入设计——而它们恰恰是最有价值的部分。

这指向一个更根本的问题：**在 AI 协作下，传统的"先设计再实现"瀑布模式是否需要重新思考？** 也许正确的模式不是 brainstorm → plan → code，而是 brainstorm → code a little → reflect → revise plan → code more，让设计和实现以更短的反馈循环交替推进。Superpowers 的流程是"先想清楚再造"，但很多工程洞察只能在造的过程中获得。

## 总结

本项目让我最受益的不是 TypeScript 技术细节（那些 Google 一下就会），而是三件事：

1. **"机制必须是代码"的判据**：它能一刀切开真正的工程工作和"给 LLM 写提示词"之间的界限，这个区别在 AI 时代会越来越重要。
2. **冷启动验证**：用不相关的 agent 审查自己的规约，是单人项目中最接近"同侪评审"的质量保障机制。
3. **工程纸面上的答案和实际编码中的答案不同**：SPEC 能回答"是什么"，但"为什么这样设计更好"的答案，只有在写了测试、跑了 CI、修了 bug 之后才能真正获得。
